# DeployPilot AI v1.0 Release Notes

## Product Summary

DeployPilot AI is an enterprise workflow automation platform for customer-facing operational teams. It helps teams onboard customers, define workflows, run AI-assisted workflow analysis, route high-risk or low-confidence work to human approval, and evaluate output quality over time.

The v1 release is intentionally practical: it focuses on a reliable demo-ready core rather than a broad feature set. It shows the full path from customer setup to workflow execution, approval, evaluation, observability, and Docker-based local deployment.

## Architecture Summary

DeployPilot AI is organized as a multi-service application:

- **Backend**: Java 21, Spring Boot 3.5, Maven, Spring Web, Spring Data JPA, Validation, Actuator, PostgreSQL.
- **AI service**: Python 3.12, FastAPI, Pydantic v2, Uvicorn, mock-safe LLM service boundaries.
- **Frontend**: Next.js and TypeScript dashboard for customers, workflows, approvals, and eval summaries.
- **Infrastructure**: Docker Compose for local orchestration with PostgreSQL, backend, AI service, and frontend.
- **Quality gates**: Unit tests, basic context tests, CI workflow, smoke-test scripts, and runbooks.

The backend owns persistent business state. The AI service owns classification, extraction, action generation, and evaluation contracts. The frontend consumes the backend through a single API wrapper configured by `NEXT_PUBLIC_BACKEND_URL`.

## Completed Features

- Customer CRUD-style APIs.
- Workflow creation and lookup APIs.
- Workflow run APIs with AI-service orchestration hooks.
- Deployment config APIs for environment-specific rollout behavior.
- Human approval APIs for pending, approve, and reject flows.
- Evaluation dataset, eval case, eval run, and eval summary APIs.
- Demo-only seed profile with EuroLogix Operations sample data.
- Docker Compose local stack.
- GitHub Actions CI for backend, AI service, frontend, and Docker build checks.
- E2E smoke test scripts for Bash and PowerShell.
- Professional frontend dashboard with loading and error states.
- Runbooks for local setup, demo data, and smoke testing.

## Demo Flow

1. Start the stack with Docker Compose.
2. Confirm backend and AI service health.
3. Create or use the demo customer: `EuroLogix Operations`.
4. Create a `PROD` deployment config.
5. Create the `SLA Breach Refund Assistant` workflow.
6. Run the workflow with a Berlin shipment delay refund request.
7. Review detected intent, extracted fields, recommended action, and run status.
8. Approve the run when it requires human review.
9. Run evals or review the eval summary.
10. Use the frontend dashboard to inspect customers, workflows, runs, approvals, and eval metrics.

## Known Limitations

- Authentication is intentionally not implemented in v1.
- AI responses are currently mock-safe and contract-focused; production OpenAI integration should be hardened behind provider interfaces, retries, and eval gates.
- Database schema management uses JPA configuration rather than formal migrations.
- The frontend is demo-ready, not a complete enterprise console.
- Approval policies are simple and should become configurable per customer, workflow, and environment.
- Observability is basic; production deployment should add structured logs, metrics, tracing, and dashboards.
